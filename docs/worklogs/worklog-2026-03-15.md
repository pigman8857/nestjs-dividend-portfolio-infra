## 13:29 — Issue 1: Hostname NXDOMAIN

**Symptom**
```
MongoNetworkError: getaddrinfo ENOTFOUND dividenddev.adb.ap-singapore-1.oraclecloud.com
```

**Investigation**
- Confirmed ADB state is `AVAILABLE` in Terraform state
- Ran `nslookup` — hostname returned `NXDOMAIN`
- Inspected `terraform show -json` → `connection_urls[0].mongo_db_url`
- Found OCI assigns: `G384A9A0B4990AA-DIVIDENDDEV.adb.ap-singapore-1.oraclecloudapps.com`
- `main.tf` was constructing: `dividenddev.adb.ap-singapore-1.oraclecloud.com` (wrong domain, no tenancy prefix)

**Fix — `main.tf`**
Replaced the manually constructed hostname with a local that reads `connection_urls[0].mongo_db_url` from the resource and extracts the actual host via regex. The URI is then built by substituting OCI's placeholder template instead of constructing it from scratch.

## 13:52 — Issue 5: `GRANT CREATE INDEX` invalid privilege

**Symptom** *(while running manual SQL in OCI Console)*
```
ORA-00990: missing or invalid privilege
```

**Investigation**
`GRANT CREATE INDEX TO MONGOAPP` triggered the error. In Oracle, `CREATE INDEX` is not a standalone grantable system privilege — it is implicit for a user's own tables once `CREATE TABLE` is granted.

**Fix — `database.tf` + `outputs.tf`**
Removed `GRANT CREATE INDEX` from the provisioner SQL and the `manual_user_sql` output.

## 13:56 — Debugging: grants not persisting in OCI SQL Worksheet

**Observation**
Each new script execution in OCI SQL Worksheet showed `User "MONGOAPP" created` at the top of the output, wiping all previously granted privileges. Only grants applied within the same script execution survived.

**Investigation**
- Checked for AFTER LOGON triggers: `OLTP_LOGON` found but `DISABLED`, body only sets parallel degree — unrelated
- Confirmed grants work within a single execution by running `GRANT + SELECT` together
- Concluded: OCI SQL Worksheet replays accumulated script history on each new execution; this is a UI behaviour, not a database issue

**Workaround**
Run all grants + `ORDS.ENABLE_SCHEMA` + verification in a single F5 (Run Script) execution without splitting across multiple runs.

## 14:40 — Issue 6: `listCollections` not authorised

**Symptom**
```
MongoServerError: Operation listCollections on NESTJS_DIVIDEND_PORTFOLIO is not authorized.
```

**Investigation**
- App connected and authenticated (Issues 1–5 resolved)
- New error on `listCollections` for database `NESTJS_DIVIDEND_PORTFOLIO`
- Root cause: Oracle ADB only permits a user to access their own schema via the MongoDB API. The MongoDB database name in the URI must match the authenticated Oracle username.
- URI had `nestjs_dividend_portfolio` as the database name; Oracle username is `mongoapp`

**Fix — `main.tf` + `variables.tf` + `outputs.tf`**
- `main.tf`: replaced `/[user]` placeholder with `adb_mongo_username` (not `mongo_db_name`) when building the URI
- `variables.tf`: removed the now-unused `mongo_db_name` variable
- `outputs.tf`: updated `nestjs_env_snippet` to emit `MONGO_DB_NAME=<username>`
- NestJS `.env`: changed `MONGO_DB_NAME` from `nestjs_dividend_portfolio` to `mongoapp`

## 16:00 — Second-round: full destroy + apply automation test

**Goal**
Validate that `terraform apply` provisions everything end-to-end without any manual SQL steps in OCI Console.

**Issue: ORDS.ENABLE_SCHEMA not executed (missing `/` PL/SQL terminator)**

`terraform apply` completed without error but NestJS still failed with the same `ORDS.ENABLE_SCHEMA` error. Confirmed `MONGOAPP` user existed in the DB — so user creation worked but `ORDS.ENABLE_SCHEMA` did not run.

Root cause: the PL/SQL block in the provisioner was missing the `/` terminator after `END;`. Oracle requires `/` on its own line to execute a PL/SQL anonymous block. Without it, ORDS buffers the block and returns HTTP 200 without executing anything.

**Issue: SQL failures invisible to Terraform**

The provisioner used `-o /dev/null` and checked only the HTTP status code. ORDS always returns HTTP 200 even when SQL fails — errors are in the response body JSON. Failed SQL was indistinguishable from success.

**Fix — `database.tf`**
- Added `/` after `END;` in the `ORDS.ENABLE_SCHEMA` block
- Replaced blind curl calls with a `run_sql` helper that captures the response body and scans for `errorCode`/`ORA-` patterns — exits 1 on SQL failure so Terraform surfaces a real error
- Added explicit `exit 1` after the ORDS readiness retry loop if ORDS never returned HTTP 200 (prevents fallthrough into SQL execution against a non-ready endpoint)
